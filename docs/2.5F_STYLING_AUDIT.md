# 2.5F Styling Audit

Purpose: evidence-first inventory of the frontend styling reality. This document lists discovered styling assets, token sources, component-level styles, inline styles, utility-like patterns, and notes on where styling authority appears fragmented. No changes or recommendations are made here — facts only.

## Key styling assets (source files)
- `frontend/src/styles.css` — primary global stylesheet with many CSS custom properties (tokens) and mode-specific overrides (`.app-shell.mode-light`, `.app-shell.mode-dark`, `.app-shell.mode-red`).
- `frontend/src/design/tokens.css` — spacing, radii, typography scale, motion tokens (CSS custom properties).
- `frontend/src/design/themes.css` — theme variable groups (`.theme-dark`, `.theme-light`, etc.) defining theme-scoped tokens.
- `frontend/src/styles/tokens.css` — a small bridging token file present in source (maps a compact token set to existing token names).
- `frontend/src/components/LocationSelector/locationSelector.css` — component-scoped CSS used by the `LocationSelector` component.
- `frontend/src/components/common/InlineExpansion.css` — component-scoped CSS for the inline expansion widget.
- `frontend/dist/assets/index-*.css` — compiled/built CSS bundle present in `dist` (contains resolved tokens, many :root variables and theme variants). This is an output artifact included here for evidence.

## Where tokens and tokens-like values appear
- Global token definitions and usage:
  - `frontend/src/styles.css` declares many CSS variables (e.g., `--gap`, `--page-bg`, `--text-primary`, `--button-bg`, and many theme overrides). This file contains the largest set of runtime-facing tokens.
  - `frontend/src/design/tokens.css` defines spacing/radius/font token primitives (`--space-*`, `--radius-*`, `--font-*`).
  - `frontend/src/design/themes.css` contains theme maps (`.theme-*`) that set `--bg-base`, `--panel-bg`, `--accent`, etc.
  - `frontend/src/styles/tokens.css` exists as a small canonical bridge mapping (e.g. `--token-space-1` → `--space-1`) — indicates an intermediate migration artifact.
  - `frontend/dist/assets/index-*.css` shows compiled CSS variables in :root and per-theme blocks (evidence of tokens being emitted into the build output).

## Component-level styling
- Several components rely on global tokens from `styles.css` and also include component-specific CSS files.
  - `frontend/src/components/LocationSelector/locationSelector.css` — local classes (`.location-selector`, `.ls-row`, `.ls-input`, `.ls-apply`, etc.).
  - `frontend/src/components/common/InlineExpansion.css` — local classes for the inline expansion UI.
  - Most other components do not have dedicated `.css` files and rely on global classes declared in `styles.css` (e.g., `.primary-decision-panel`, `.module-shell`, `.dashboard`, `.panel`, etc.).
  - The `dist` CSS contains many class selectors for components and modules indicating the build output is a flattened combination of global + component styles.

## Inline styles and runtime style expressions
- A measurable number of components (representative examples below) use inline `style={{ ... }}` in JSX/JS files:
  - `frontend/src/components/Conditions.jsx` — inline styles for flex layout, badges, and color usage (e.g., `style={{ padding: '6px 10px' }}`, `style={{ color: 'var(--text-muted)' }}`).
  - `frontend/src/components/RecommendedTargets.jsx`, `TargetRow.jsx`, `SatellitePasses.jsx`, `AlertsEvents.jsx` — inline layout styles for lists and image sizing, e.g. `style={{ width: 48, height: 48, objectFit: 'cover', borderRadius: 6 }}`.
  - `frontend/src/components/common/InlineExpansion.jsx` — toggles content via inline `style={{display: open ? 'block' : 'none'}}`.
  - `frontend/src/components/ui/GlassPanel.jsx` and several `ui/*` components use inline style objects for dynamic visual effects.

## Utility-like patterns and conventions observed
- CSS custom properties (variables) are used heavily as the canonical runtime token vehicle in `styles.css` and compiled `dist` output.
- There is a de facto pattern of module classes (e.g., `.module-shell`, `.primary-decision-panel`, `.panel`) that components reuse.
- Some components embed inline styles for small layout or image sizing differences rather than consuming a shared utility class.
- There exists a small token bridge (`src/styles/tokens.css`) that appears designed to ease migration — suggests tokens are currently split across multiple files.

## Evidence of hybrid or fragmented authority
- Multiple token sources: `styles.css`, `design/tokens.css`, `design/themes.css`, and `styles/tokens.css` all define or reference overlapping tokens/variables. This indicates more than one place where token values can be authored.
- Component-local CSS files exist (e.g., `locationSelector.css`) while most layout and theme tokens live in a global `styles.css` — mixing global tokens with local hard-coded CSS in places.
- Inline style usage (JSX `style` props) coexists with CSS variables and rule-based classes.
- The compiled `dist` asset contains :root and `.theme-*` blocks as well as many class selectors — this is an artifact of the build and confirms that runtime token values are emitted, but does not clarify single-source authority in source files.

## Risk summary (concise, evidence-based)
- Fragmentation risk: tokens and visual authority are present in multiple source files (`styles.css`, `design/tokens.css`, `design/themes.css`, and `styles/tokens.css`). Without a clear canonical single-source-of-truth, future edits may diverge across these locations.
- Hybrid usage: presence of inline styles plus component CSS files plus global theme tokens increases the chance of inconsistent visual updates and accidental hybridity.
- Discovery risk: compiled `dist` contains many resolved variables and component rules — good for runtime evidence but not a single authoring surface in the source tree.
- Mitigation (note: no recommendation here) — the audit shows where a canonical decision will need to point (one token file and one styling pattern) before migrations begin.
