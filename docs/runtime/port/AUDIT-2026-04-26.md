The current Sky Engine has real implementation work in these areas:

- HiPS/EPH/tile selection, HEALPix, file-backed tile repository, and deterministic module1 replay.
- Module2 stars seams including B-V color, nuniq decode, HIP lookup, render visitor traversal, catalog astrometry, apparent-place projection chain, stars list traversal, survey registry, label/designation adapters, frame-pacing traces, and an initial side-by-side harness.

This is meaningful progress, but it is not yet indistinguishable from Stellarium.

## Usability And Performance Finding

The current runtime is not purely CPU-rendered. `directStarLayer.ts` uses Babylon thin instances and WebGL buffers.

The remaining performance risk is that the runtime still does large CPU-side work before each GPU draw:

- star tile traversal and projection in `runtimeFrame.ts::collectProjectedStars`
- object assembly and query/model sync in `SkyEngineScene.tsx`
- per-star matrix/color buffer fill in `directStarLayer.ts`
- buffer update/upload through Babylon thin-instance buffers

Stellarium also prepares render items on CPU, but its central path runs through `core.c -> painter.c -> render_gl.c`, with renderer-owned batching, clip state, shader state, point/mesh buffers, tonemapping, and paint lifecycle. Those source files are not ported into the Hub runtime.

Therefore the current lag/usability problem should not be documented as "no GPU." It should be documented as:

```text
GPU draw exists, but the Stellarium painter/render_gl batching, clipping, shader, tonemapper, and live side-by-side frame lifecycle are not ported.
```

## Critical Drift Corrections

The port docs must prevent these false impressions:

1. `module0` / `module1` complete does not mean their native source files are fully ported.
2. `module2` test growth does not mean the visible runtime is Stellarium-equivalent.
3. Hub-only deterministic replay does not close side-by-side parity.
4. `painter.c`, `render_gl.c`, `shader_cache.*`, `tonemapper.*`, `texture.*`, `gl.*`, and frame lifecycle code are not optional late polish; they are required for minimal Stellarium-indistinguishable behavior.
5. The `module7-remaining-swe` bucket is too coarse for a full port and must be split before claiming port readiness beyond module2.

## What Is Left

Immediate module2 blockers:

- G5 side-by-side checkpoints must be sourced from the live pinned Stellarium runtime, not committed Hub-only fixture vectors.
- `stars_list` / `stars_add_data_source` must close against live fetch lifecycle, not only loaded fixture and adapter seams.
- Runtime profiling must be tied to active-scene telemetry and a source-backed rendering plan.

Full-port blockers after module2:

- `painter.c` / `painter.h` paint lifecycle, clipping, point/mesh/text/texture contracts.
- `render_gl.c` / `render.h` renderer item batching and GL draw path.
- `core.c` full update/render lifecycle beyond the current preamble seams.
- `module.c` object/module lifecycle and sort/update/render/post-render semantics beyond the current `SkyModule` approximation.
- `projection.c` and all projection implementations at source-contract level.
- `observer.c`, `frames.c`, and navigation functions at native side-by-side parity, not only ERFA/PyERFA subset parity.
- UI modules and overlays (`labels`, `skycultures`, `constellations`, `lines`, `cardinal`, `pointer`) at source behavior level.
- Planets, atmosphere, landscape, DSO/DSS, satellites, comets/minor planets/meteors, and remaining utility/runtime systems.
