# PHASE 2 — ENGINE EXPLORATION

## 1. Purpose

Phase 2 expands Astronomy Hub from a single-scene system into a **multi-engine exploration system**.

The system must now answer:

> “What should I know about the sky right now, across different domains, and what should I look at?”

Phase 2 introduces **engine-level exploration** while preserving the Phase 1 decision model.

---

## 2. Phase 2 Definition

### 2.1 What Phase 2 IS

Phase 2 is:

- Expansion from one scene (“Above Me”) to **multiple engines**
- Introduction of **scope → engine → filter → scene → object → detail**
- Structured exploration across domains:
  - sky
  - satellites
  - solar system
  - deep sky
  - atmospheric/earth context

- Controlled filtering within each engine
- Multiple scene entry points, each tied to an engine

Phase 2 enables users to explore **different observational contexts** without breaking system structure.

---

### 2.2 What Phase 2 is NOT

Phase 2 is NOT:

- A UI redesign phase
- A visualization phase (no maps, no 3D, no charts)
- A prediction or AI phase
- A personalization phase
- A freeform search system
- A data explorer

Phase 2 MUST NOT:

- Replace Phase 1 behavior
- Introduce new rendering paradigms
- Break scene → object → detail flow

---

## 3. System Model

Phase 2 formalizes the system pipeline:

```

User → Scope → Engine → Filter → Scene → Object → Detail

```

### Definitions

- **Scope**  
  Top-level context (e.g., sky, earth, solar system)

- **Engine**  
  Domain-specific system (e.g., satellites, planets, deep sky)

- **Filter**  
  Constraints applied within an engine (e.g., visible now, bright only)

- **Scene**  
  A decision surface generated by an engine under filters

- **Object**  
  A selectable entity within a scene

- **Detail**  
  Authoritative object information

---

### 4.1 Allowed Scopes

- sky
- solar_system
- earth

Scopes define which engines are available.

## 5. Engines

### 5.1 Required Engines

#### Sky Scope
- above_me (existing)
- deep_sky

#### Solar System Scope
- planets
- moon

#### Earth Scope
- satellites
- flights (if implemented, optional)

---

### 5.2 Engine Responsibilities

Each engine MUST:

- Generate scenes
- Select objects
- Apply filters
- Provide reasoning for object inclusion

Each engine MUST NOT:

- Bypass the scene model
- Return raw data without structure
- Duplicate object detail logic

---

## 6. Filters

Filters are introduced in Phase 2 as controlled constraints.

### 6.1 Examples

- visible_now
- bright_only
- high_altitude
- short_window
- naked_eye

### 6.2 Rules

Filters MUST:

- be engine-specific where necessary
- affect object inclusion and grouping
- not exist as freeform user queries

Filters MUST NOT:

- create new scene types
- bypass engine logic

---

## 7. Scenes

Scenes remain the **primary decision unit**.

### 7.1 Scene Behavior

Each scene MUST:

- Answer a specific decision question
- Contain grouped objects
- Provide reasoning
- Reflect current conditions and time

### 7.2 Scene Types

Scenes are generated per engine and filter combination.

Examples:

- Above Me (sky engine)
- Deep Sky Visible Now
- Planets Visible Tonight
- Satellite Passes Soon

Scenes are NOT freeform.

---
## 7.3 Scene Output Contract (Phase 2 Clarification)

This section defines the **minimum required structure and rules** for all scene outputs in Phase 2.

This is NOT a duplicate of DATA_CONTRACTS.md.  
It defines **behavioral requirements and required fields for scene validity**.

---

### 7.3.1 Scene Structure Requirements

Every scene MUST include:

- a clear title
- a short summary explaining what the user should understand
- a collection of grouped objects
- reasoning for surfaced objects
- time relevance context
- observing context (conditions)

Scenes MUST NOT:

- return raw lists of objects
- rely on frontend interpretation for meaning
- omit reasoning

---

### 7.3.2 Object Requirements (Within Scenes)

Every object surfaced in a scene MUST include:

- identity (id)
- name
- object type (planet, galaxy, satellite, etc.)
- visibility context (e.g., visible, low, obstructed)
- time relevance (e.g., now, soon, setting, peak later)
- reason for inclusion

Example reasoning:

- “High altitude and easy to observe”
- “Short viewing window — setting soon”
- “Peak visibility tonight”

Objects MUST NOT:

- include full detail payloads
- exist without a reason
- appear if not actionable

---

### 7.3.3 Grouping Requirements

Objects MUST be grouped.

Each group MUST:

- have a title
- represent a meaningful category
- include a reason for grouping

Examples:

- “Best Targets Right Now”
- “Setting Soon”
- “High Visibility”
- “Short Window Opportunities”

Groups MUST NOT:

- be arbitrary
- be purely visual
- duplicate each other

---

### 7.3.4 Reasoning Requirement (Critical)

Every scene MUST provide reasoning at two levels:

1. Scene-level summary  
2. Object-level reasoning  

This ensures the system answers:

> “Why should I care about this?”

---

### 7.3.5 Time Context Requirement

Scenes MUST reflect time in a human-meaningful way.

Required concepts:

- now (current relevance)
- soon (near-future relevance)
- best window (peak visibility if applicable)

Time expressions must be understandable without technical interpretation.

---

### 7.3.6 Conditions Requirement

Scenes MUST reflect observing conditions.

At minimum:

- visibility quality
- moon impact
- general observing notes (if relevant)

Conditions MUST:

- influence object selection
- influence grouping
- influence reasoning

Conditions MUST NOT:

- exist as a separate scene
- be ignored by engines

---

### 7.3.7 Authority Rule

The backend is the sole authority for:

- object inclusion
- grouping
- reasoning
- prioritization

The frontend MUST NOT:

- reorder results arbitrarily
- generate reasoning
- filter objects independently

---

### 7.3.8 Valid Scene Definition

A scene is valid ONLY if:

- it answers a decision question
- it contains grouped objects
- it includes reasoning
- it reflects time and conditions
- it can stand alone without object detail

If any of these are missing, the scene is invalid.

---

## 8. Objects

Objects are:

- surfaced by scenes
- selected by the user
- resolved through `/api/object/{id}`

### 8.1 Object Rules

Objects MUST:

- include visibility context
- include time relevance
- include reason for inclusion

Objects MUST NOT:

- contain full detail payloads
- exist outside scene context

---

## 9. Backend Requirements

Phase 2 backend MUST:

- support multiple engines
- support filters per engine
- produce scene-compliant outputs
- maintain object-detail authority
- avoid duplication of logic

### Required Additions

- engine routing layer
- filter handling layer
- multi-scene endpoint support

---

## 10. Frontend Requirements

Frontend MUST:

- allow scope selection
- allow engine switching
- allow filter selection
- render scenes consistently
- preserve object → detail flow

Frontend MUST NOT:

- interpret or modify scene logic
- generate its own rankings
- duplicate backend reasoning

---

## 11. Interaction Model

### Phase 1:
User → Scene → Object → Detail

### Phase 2:
User → Scope → Engine → Filter → Scene → Object → Detail

The system must remain **guided**, not exploratory chaos.

---

## 12. Constraints

### 12.1 Architecture Constraints

- Backend is authoritative
- Scene is the decision surface
- Object detail is isolated

### 12.2 Complexity Constraints

- No duplication of engine logic
- No custom rendering per engine
- No uncontrolled expansion

---

## 13. Anti-Scope Rules

The following are strictly prohibited in Phase 2:

- 3D sky maps
- sky globe visualizations
- timeline scrubbing
- predictive recommendations
- AI assistants
- social features
- astrophotography tools
- free search systems
- UI redesign efforts

---

## 14. Completion Criteria

Phase 2 is complete ONLY when:

- multiple engines exist and function
- scope → engine → filter → scene flow works
- scenes remain decision-driven
- object detail remains authoritative
- no scope drift has occurred
